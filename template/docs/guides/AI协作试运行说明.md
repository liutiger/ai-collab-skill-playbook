# AI 协作试运行说明

> 版本：v1.0
> 适用时间：第一批试运行
> 适用对象：明天开始直接在 VS Code + GitHub Copilot Chat 里使用 AI 协作的研发同学

## 1. 这份文档是干什么的

这不是总纲，也不是 Skill 设计文档。

这份说明只解决一件事：

**明天开始，团队成员怎么低成本、低风险地把 AI 用起来。**

如果你只想先用，不想先理解整套 Prompt / Skill 实现，请只看这份文档即可。

若需要回查原则与边界，再看：

- `docs/guides/AI协作研发章程.md`
- `docs/guides/ai-workflow.md`

## 2. 先记住 3 条规则

### 规则一：先分场景，再分阶段

默认先用 `/wms-scene-router`，它会结合 `12-scene-catalog.md` 判断这次到底是：

- 新功能开发
- 线上排障
- 代码审查
- 数据库变更
- 重构优化
- 文档编写

如果它判断出来是“确认真实链路”，就直接转 `/wms-link-trace`。

### 规则二：边界不清或风险高时，先用 `/wms-plan-gate`

下面这些情况，一律先用 `/wms-plan-gate`，不要直接让 AI 写代码：

- 需求边界不清
- 涉及跨模块、跨服务
- 涉及公共接口、核心链路
- 你自己也不确定怎么改

### 规则三：能让 AI 写代码，不代表应该直接写

只有在以下条件满足后，才进入实现：

- 目标已经说清楚
- 方案你已经认可
- 风险点你能接受
- 改动范围基本可控

这时再进入：

- `/wms-auto-dev`

### 规则四：改完代码不等于完成，最后还要过评测门禁

凡是这些内容，不要只留在聊天框里：

- 方案结论
- 链路梳理
- 根因分析
- 改动结果
- 测试结论

要沉淀到任务目录里；涉及代码 / SQL / 配置改动时，完成前还要补一轮 `/wms-evaluation-gate`。

## 3. 最小使用路径

### 第一步：先建任务目录

在项目根目录执行：

```bash
./docs/skills-src/wms/wms-task-governance/scripts/create_task_dir.sh \
  --owner "{你的名字}" \
  --module "{模块名}" \
  --plan-gate \
  --evaluation-gate \
  "{任务名}"
```

如果你已经知道要做链路确认，再补 `--link-trace`。如果本轮没有代码 / SQL / 配置改动，可去掉 `--evaluation-gate`。

### 第二步：在 Copilot Chat 里选对入口

推荐优先级：

1. 先判断任务属于哪个场景：`/wms-scene-router`
2. 不清楚、风险高、想先核对方案：`/wms-plan-gate`
3. 方案已确认，想让 AI 走完整实现闭环：`/wms-auto-dev`
4. 改完后要决定能不能算完成：`/wms-evaluation-gate`
5. 想确认真实业务链路、副作用、有没有真正落盘：`/wms-link-trace`

### 第三步：把任务说清楚

聊天框里不要只说“帮我看下”，尽量说明：

- 要做什么
- 范围在哪
- 验收标准是什么
- 已知日志 / 现象 / 限制是什么

一个可直接使用的例子：

```text
/wms-scene-router

目标：新增库存批量操作类型能力
范围：example-service、同步链路
验收：能新增类型且不影响现有库存扣减
限制：请先判断主场景，并告诉我下一步该走哪个入口
```

### 第四步：确认方案后再进入实现

如果上一轮已经完成场景判断和方案确认，再切到：

- `/wms-auto-dev`

并明确说明：

- 方案已确认
- 允许开始实现
- 改完要自检、测试、评测门禁、落盘

### 第五步：代码改完后补评测门禁

如果本轮有代码 / SQL / 配置改动，再进入：

- `/wms-evaluation-gate`

目标不是再讨论一遍，而是明确：

- 跑了哪些门禁
- 结果是 `PASS / CONDITIONAL_PASS / BLOCKED / FAIL`
- 是否允许写完成标记

## 4. 明天试运行时只要求做到这些

第一版先不要追求“全自动、全无缝、全靠 Skill 自动路由”。

先把以下最小动作跑通：

- 能先分对场景，再让阶段入口自动带上专项检查
- 能在高风险任务上先停在方案阶段
- 能避免一上来全项目乱搜
- 能把关键结论写进 `docs/tasks/`
- 能在代码改动后补评测门禁并明确结果

只要这几件事稳定，第一版就是成功的。

## 5. 推荐使用习惯

### 推荐这样做

- 先用 `/wms-scene-router` 分场景，再决定要不要进 `/wms-plan-gate`
- 复杂任务拆成两轮：先方案，后实现
- 改公共接口、核心方法前先看影响面
- 任务结束前回看 `docs/tasks/` 是否已经落盘，评测门禁是否已完成

### 不推荐这样做

- 一上来就让 AI “直接写完”
- 跳过方案确认直接改跨模块逻辑
- 只在聊天框里得到结论，不沉淀
- 连场景都没分清就直接进 `/wms-auto-dev`

## 6. 什么时候要停下来找人确认

出现下面情况时，不要让 AI 继续自动推进：

- 有两个以上方案，取舍明显不同
- 需要改公共接口或数据库结构
- 需要跨服务联动
- 你读完方案后依然说不清“为什么这么改”
- AI 的定位和你的业务理解明显冲突

这时先人工确认，再继续。

## 7. 文档关系怎么理解

第一版请这样理解，不要把文档层级混掉：

- `AI协作研发章程.md`：总纲，讲原则和边界
- `AI协作试运行说明.md`：给明天就要开始用的人
- `ai-workflow.md`：完整流程说明
- `docs/skills-src/`：维护者才需要关注的实现源码
- `.github/prompts/`：Copilot 的实际入口

## 8. 一句话记忆

**先分场景，再定阶段；先出方案再实现，代码改完还要过评测门禁。**
