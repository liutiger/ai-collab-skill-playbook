# AI 协作研发章程

> 版本：v1.0
> 适用项目：`example-wms-project`
> 说明：本文档是公开模板中的示例总纲，使用时请按自己的项目、角色和风险边界重写
> 适用对象：使用 AI 参与需求分析、代码开发、问题排查、代码审查、数据库变更、链路确认和知识沉淀的研发成员

## 1. 文档定位

本章程不是某一条提示词的说明书，而是本项目使用 AI 参与研发时的总纲。  
它回答 4 个问题：

1. 我们为什么要让 AI 参与研发
2. AI 在研发流程中的定位是什么
3. AI 必须遵守哪些工作方式和控制边界
4. AI 的产出如何沉淀成团队长期资产

本章程与以下内容配合使用：

- 项目上下文：`AGENTS.md`
- 试运行入口：`docs/guides/AI协作试运行说明.md`
- AI 工作流：`docs/guides/ai-workflow.md`
- 提示词体系：`docs/prompts/00~19`
- 任务沉淀目录：`docs/tasks/`
- 长期知识库：`docs/knowledge-base/`

### 1.1 治理引用顺序

本项目所有 AI 研发协作资产遵循以下引用顺序：

1. `AI协作研发章程`：总纲，定义定位、边界、原则和完成标准
2. `AI 工作流与试运行说明`：把章程落实成团队标准用法
3. `docs/skills-src/`：Skill 源码与生成规范
4. `Prompt / Copilot slash prompt / IDE 适配层`：把工作方法适配到具体工具入口
5. `单次任务输入`：描述这次任务的目标、范围和约束

这 5 项是治理引用顺序，不是另一套独立架构。真正的运行架构定义，以后文的 **双轴五构件** 为准。

> `TDD / ACD / AOD` 是 **Orchestrator 之下的 Strategy Packs**，不是新增架构层级。

任何 Skill、Prompt、Copilot 适配层都不得脱离本章程单独演化；若与本章程冲突，以本章程为准。

### 1.2 阅读顺序

本章程是治理总纲，不是给一线研发同学的“明天直接上手手册”。

不同角色的推荐阅读顺序如下：

- 日常使用 AI 协作的研发同学：先看 `docs/guides/AI协作试运行说明.md`，再按需回查本章程
- 需要理解完整流程的同学：先看 `docs/guides/AI协作试运行说明.md`，再看 `docs/guides/ai-workflow.md`
- 维护 Prompt / Skill / Copilot 适配层的人：先看本章程，再看 `docs/guides/AI协作提示词Skill化方案.md` 和 `docs/skills-src/README.md`

对于第一版试运行，默认原则是：

- 先保证团队会用、敢用、不会明显跑偏
- 再逐步扩大 Skill、脚本和自动化覆盖面
- 不要求一线同学先理解 `docs/skills-src/`、`.claude/skills/` 等实现细节

## 2. 宗旨

AI 进入研发流程的目标，不是替代工程判断，而是提升复杂项目中的分析效率、交付稳定性和知识沉淀质量。

在本项目中，AI 的使用必须同时满足以下目标：

- 提高复杂链路任务的处理效率
- 降低跨模块分析和改动时的遗漏率
- 减少纯人工检索、纯人工整理带来的时间浪费
- 将一次性对话转化为可复用的团队知识
- 让 AI 的行为可追溯、可审查、可复盘、可接续

## 3. 适用范围

本章程适用于以下场景：

- 新功能开发
- 线上问题排查
- 代码审查
- 数据库变更与 SQL 评审
- 重构优化
- 文档编写
- 链路确认
- 方案核对与实施前评审
- 自动评测与交付验收

本章程不适用于以下行为：

- 将 AI 作为无记录的临时聊天工具使用
- 跳过项目规范，直接要求 AI 大范围改代码
- 没有任务目录、没有文档闭环的“口头式协作”

## 4. 核心定义

### 4.1 AI

指参与本项目研发活动的模型、Agent、IDE 内 AI 助手、命令行 AI 工具及其组合。

### 4.2 Prompt

Prompt 的职责是定义本次任务的目标、范围、场景和约束。  
Prompt 主要解决“这次任务要做什么”。

### 4.3 Skill

Skill 的职责是固化一类任务的稳定工作方法、工具协议、输出契约和必要脚本。  
Skill 主要解决“这类任务稳定地应该怎么做”。

### 4.4 Workflow

Workflow 指一项任务从启动、分析、定位、设计、实现、验证到沉淀的完整协作闭环。

### 4.5 Knowledge Sink

Knowledge Sink 指任务过程中必须写入的沉淀位置，包括：

- `docs/tasks/`
- `docs/knowledge-base/`
- `artifacts/`
- `AGENTS.md`（项目级共识）

### 4.6 双轴五构件

为避免能力在“统一收敛”时被错误吞并，本模板把 AI 协作体系定义成 **双轴五构件**，而不是“很多层并排堆叠”的模型。

#### Governance Axis

- `Charter`：总纲，定义定位、边界、原则和完成标准
- `Governance Contract`：共享生命周期合同，例如规划标记、结束前确认、多轮补充检查

#### Execution Axis

- `Orchestrator`：默认总控入口，统一判主场景、选择执行策略、推进 `DISCOVER / DELIVER / VERIFY` 三态，并按需叠加专项方法
- `Capability Packs`：按能力维度组织的事实来源，包括 `Scene Packs`、`Strategy Packs`、`Method Packs`、`Gate Packs`
- `Adapters & Tools`：把工作法适配到 Copilot / Skill 运行产物、任务初始化脚本和生成验收工具

其中：

- `Scene Packs` 负责回答“这次任务属于什么类型”
- `Strategy Packs` 负责回答“这轮怎么推进”，`TDD / ACD / AOD` 就属于这一类
- `Method Packs` 负责回答“是否要叠加专项工作方法”，例如链路确认
- `Gate Packs` 负责回答“是否允许进入验证或完成”，例如评测门禁

## 5. AI 在本项目中的定位

AI 不是单纯的问答工具，也不是可以无限自由发挥的黑盒助手。

在本项目中，AI 的定位是：

- 研发协作者
- 结构化分析者
- 多角色执行代理
- 文档沉淀参与者

AI 不是以下角色：

- 最终业务裁决者
- 架构变更的唯一决策者
- 无需审查即可直接落地的自动提交者

最终业务判断、实施批准、风险取舍，仍由人负责。

## 6. 基本原则

### 6.1 先证据，后结论

AI 的结论必须基于代码、配置、调用链、日志、文档等可验证证据，不允许凭经验脑补关键结论。

### 6.2 先定位，后修改

在涉及代码理解、链路确认、风险评估、修改建议时，必须先完成模块、符号、调用链和影响面的定位。

### 6.3 先方案，后实现

高风险任务、跨模块任务、链路复杂任务，必须先进入方案阶段，经确认后再进入实现阶段。

### 6.4 先最小改动，后大范围重构

AI 默认优先选择最小安全改动路径，不得将一次局部需求扩展成无边界重构。

### 6.5 先沉淀，再结束

任务完成不以“回答了用户问题”为标准，而以“分析、实现、验证和知识沉淀已完成”为标准。

### 6.6 先收敛，再扩散

AI 处理复杂问题时，应先缩小边界、确认入口、聚焦候选模块，再逐步扩展，不允许一上来全项目搜索。

## 7. 导航与检索章程

### 7.1 工具优先级

涉及代码定位、调用链分析、影响评估时，固定顺序为：

`GitNexus > 局部 rg > 全项目搜索`

### 7.2 标准导航顺序

1. `status`：确认 GitNexus 索引是否可用
2. `query`：按业务词、异常词、接口名寻找流程和模块
3. `context`：查看候选符号的调用者、被调用者、参与流程
4. `impact`：在修改公共符号或核心方法前确认影响范围
5. 只有图谱无法覆盖时，才补充文本检索

### 7.3 禁止行为

- 不允许一上来全仓库乱搜
- 不允许把多个猜测方法名、动作词拼成一条长正则碰运气
- 不允许在未确认目标模块前随机翻大文件多个区间
- 不允许在连续两轮定位失败后继续扩大关键词，而不重写假设

### 7.4 目标

导航的目标不是“搜到点东西”，而是回答以下问题：

- 入口在哪
- 主编排在哪
- 关键分支在哪
- 关键副作用在哪
- 改动会波及谁

## 8. 标准协作流程

### 8.1 任务启动

- 获取最新规范
- 确认 GitNexus 可用性
- 创建任务目录
- 写入 README 初始信息与规划标记
- 对续做任务检查已完成和待处理边界

### 8.2 分析阶段

- 澄清目标、范围、验收标准、非目标
- 明确当前任务属于哪个场景
- 用 GitNexus 完成模块、符号、调用链、影响面定位
- 提炼关键假设和待确认点

### 8.3 设计阶段

- 给出最小改动方案
- 列出备选方案
- 明确风险点、回滚思路和验证方式
- 高风险任务需人工确认

### 8.4 实施阶段

- 按项目现有架构和命名边界实施改动
- 优先小步、局部、可解释的实现
- 涉及核心链路时回看影响分析

### 8.5 验证阶段

- 检查架构合规
- 检查业务正确性和边界条件
- 明确已执行和建议执行的验证动作
- 标记残留风险

### 8.6 评测与验收阶段

涉及代码 / SQL / 配置改动时，必须进入自动评测与交付验收门禁：

- 明确影响模块和评测范围
- 至少执行编译门禁和测试门禁，按风险补充静态门禁
- 产出结构化评测结论（`PASS / CONDITIONAL_PASS / BLOCKED / FAIL`）
- `BLOCKED` 或 `FAIL` 时，不得写完成标记

### 8.7 沉淀阶段

- 写入 README
- 写入 `ai-conversations/`
- 写入 `business-insights.md`
- 写入 `artifacts/`
- 对可复用认知同步到 `knowledge-base/`
- 最后追加完成标记

## 9. 角色职责

### 9.1 Planner

- 澄清目标、范围、非目标、验收标准
- 明确关键假设和待确认点
- 防止任务一开始就发散

### 9.2 Context Analyst

- 通过 GitNexus、必要的局部检索和文档定位真实上下文
- 确认模块、符号、流程、影响面

### 9.3 Solution Architect

- 设计最小改动方案
- 给出风险和回滚路径
- 明确验证方式

### 9.4 Implementer

- 按已确认方案实施改动
- 遵守分层、接口归属、模块边界和已有模式

### 9.5 Reviewer & Tester

- 做架构、自检、正确性、事务、并发、幂等和风险检查
- 给出验证结果或验证建议

### 9.6 Evaluator / Gatekeeper

- 明确评测范围、执行门禁和门禁结论
- 决定当前任务是否允许进入完成态
- 对 `BLOCKED / FAIL / CONDITIONAL_PASS` 给出透明说明

### 9.7 Documenter / Knowledge Curator

- 将任务结论沉淀到 `docs/tasks/`
- 将可复用知识沉淀到 `knowledge-base/`
- 在必要时推动更新 `AGENTS.md`

## 10. 场景治理

本项目不使用“一套 Prompt 处理所有任务”的方式，而按场景切换工作协议。

当前标准场景包括：

- `01` 新功能开发
- `02` 线上问题排查
- `03` 代码审查
- `04` 数据库变更
- `05` 重构优化
- `06` 文档编写
- `07` 多角色自动开发编排
- `08` 链路确认 / 知识沉淀
- `09` 方案核对关卡
- `10` 自动评测 / 交付验收门禁

场景切换的目的不是切换“人格”，而是切换：

- 检查重点
- 输出格式
- 风险控制方式
- 产出物要求

## 11. 文档沉淀章程

### 11.1 对话必须沉淀

凡是产生了以下内容之一的 AI 对话，均应沉淀：

- 可采纳的方案
- 有价值的分析
- 真实的链路梳理
- 代码改动建议或结果
- 根因分析
- 数据修复方案

### 11.2 沉淀层级

- 单次任务过程：`docs/tasks/`
- 可复用业务知识：`docs/knowledge-base/business-flows/`
- 架构机制与模式：`docs/knowledge-base/architecture/`
- API 契约：`docs/knowledge-base/api/`
- 数据结构与表关系：`docs/knowledge-base/database/`

### 11.3 完成定义

满足以下条件，任务才算真正完成：

- 场景目标达成
- 核心风险已说明
- 关键验证已完成或明确给出
- 评测门禁结论已给出
- 任务文档已更新
- 完成标记已追加

## 12. 风险控制章程

### 12.1 人工确认关卡

以下情况必须在方案阶段停下，待人工确认：

- 需求边界不清
- 影响范围较大
- 需要跨模块或跨服务改动
- 需要改变公共接口、核心链路或关键状态流转
- 方案之间存在明显取舍

### 12.2 影响面报告

涉及公共符号、核心类、核心方法、接口定义改动时，必须输出：

- 目标符号
- 主要调用方
- 风险级别
- 是否建议灰度

### 12.3 残留风险透明化

AI 不允许用模糊话术掩盖不确定性。  
凡是不确定、未验证、依赖推断的部分，必须明确写出。

### 12.4 未过评测门禁不得完成

若任务涉及代码 / SQL / 配置改动，则：

- 未生成 `eval-report` 不得视为完成
- 关键门禁为 `BLOCKED` 或 `FAIL` 不得写完成标记
- `CONDITIONAL_PASS` 必须明确条件和残留风险

## 13. 反模式

以下行为视为违反本章程：

- 未引用部门规范直接执行任务
- 未创建任务目录即开展多轮 AI 协作
- 跳过 GitNexus 直接进行大范围搜索
- 未分析影响面就修改公共符号
- 跳过方案确认直接实施高风险改动
- 未经过评测门禁就把代码改动视为已完成
- 有价值结论未沉淀到 docs
- 写入完成标记前未回查新增任务
- 把 AI 的一次输出当作最终事实而不做证据核对

## 14. 治理演进方向

当前项目已经具备较完整的 Prompt 体系，但长期方向不是继续无限增加 Prompt，而是逐步升级为：

- 项目上下文层
- 核心工作方法层
- 场景入口层
- 脚本与模板层
- 自动化与定时任务层

也就是从“提示词体系”演进为“可安装、可触发、可复用、可审计的 Skill 体系”。

对应方案见：

- `docs/guides/AI协作提示词Skill化方案.md`

---

## 15. 一句话总纲

本项目的 AI 协作研发，遵循以下总原则：

**以 GitNexus 为导航，以多角色分阶段执行为骨架，以评测门禁和文档沉淀为闭环，以人工确认和风险控制为护栏。**
