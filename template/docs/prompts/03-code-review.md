# 提示词 03 — 代码审查场景检查包

> 对话开头可引用 `#file:docs/prompts/03-code-review.md`。
> 更推荐先走 `#file:docs/prompts/11-scene-router.md` 判定场景；若确认为 Review，再使用本检查包。
>
> **兼容入口**：本场景的权威工作法以 `code-review-scene` 为准。
> 这是场景检查包，不承担完整共性流程。

---

## 何时使用

- 用户给出 diff、PR、代码片段，要求做 Review
- 目标是发现风险、回归点、遗漏测试，而不是直接帮忙实现
- 需要按严重度输出 findings

## 不用于

- 需求尚未形成代码：先走 `09-plan-gate.md`
- 用户明确要求直接修改代码：先确认是否转入 `07-auto-dev-orchestration.md`
- 只是梳理链路：改走 `08-link-confirmation.md`

---

## 场景判定信号

- 常见用户表达：`帮我 review`、`看下这个 PR`、`这个改动有没有问题`
- 常见产出：阻断级问题、警告级问题、风险说明、测试缺口

---

## 专项检查重点

### 1. Findings 优先

- 先列问题，再给总结
- 按严重度排序，而不是平均铺开
- 尽量给到类 / 方法 / 行位置信息

### 2. 正确性与一致性

- 业务状态流转
- 事务、幂等、并发、消息顺序
- 空值、边界值、失败回滚

### 3. 架构和安全

- 分层是否越界
- 公共接口是否合规
- SQL 注入、日志泄露、反序列化等风险

### 4. 测试缺口

- 哪些测试缺失会导致回归风险
- 哪些改动应该补单测 / 集成测试 / 链路回归

---

## 默认产出物

- `docs/reports/code-review/...-review.md`
- `docs/tasks/.../README.md`（关联任务时）
- `knowledge-base/architecture/`（沉淀新反模式时）

---

## 推荐组合

- 直接做审查：`00 + 11 + 03`
- Review 后决定要修：`00 + 11 + 09` 或 `00 + 11 + 07`

---

## 场景自检

1. [ ] 输出以 findings 为主，不是先写大段总结
2. [ ] 已覆盖正确性、并发、事务、安全和可维护性
3. [ ] 关键公共符号已结合影响面或上下文看风险
4. [ ] 未把“我来顺手改掉”混进 Review 结论里
5. [ ] 需要沉淀的反模式会写入报告或知识库
