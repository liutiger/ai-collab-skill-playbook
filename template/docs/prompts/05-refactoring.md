# 提示词 05 — 重构优化场景检查包

> 对话开头可引用 `#file:docs/prompts/05-refactoring.md`。
> 更推荐先走 `#file:docs/prompts/11-scene-router.md` 判定场景，再叠加 `09-plan-gate.md` 或 `07-auto-dev-orchestration.md`。
>
> **兼容入口**：本场景的权威工作法以 `refactoring-scene` 为准。
> 这是场景检查包，不承担完整共性流程。

---

## 何时使用

- 目标是改善结构、降低复杂度、拆分职责、提升可维护性
- 要求默认保持行为不变
- 需要先判断风险等级、提交切分和回滚路径

## 不用于

- 借“重构”名义顺手改业务逻辑
- 实际是新功能开发或线上修复
- 只做代码审查，不做改动

---

## 场景判定信号

- 常见用户表达：`重构`、`拆方法`、`抽公共逻辑`、`行为不变`
- 常见产出：风险分级、分步方案、回滚思路、重构分析报告

---

## 专项检查重点

### 1. 风险分级

- 高风险：核心链路、公共接口、并发 / 事务敏感逻辑
- 中风险：主流程 Service 编排、跨模块协作
- 低风险：工具类、命名、局部结构优化

### 2. 行为不变

- 不混入需求变更
- 不顺手修 unrelated bug
- 先明确“什么不能变”

### 3. 提交切分

- 拆成可回滚的小步
- 每一步都最好可独立验证
- 影响面大的先看上下文和 blast radius

### 4. 验证与回滚

- 至少说明行为验证点
- 核心链路重构后默认进入评测门禁

---

## 默认产出物

- `artifacts/refactoring-analysis.md`
- `artifacts/eval-report.md`
- `knowledge-base/architecture/`

---

## 推荐组合

- 先判断值不值得改、怎么拆：`00 + 11 + 09 + 05`
- 确认后分步实现：`00 + 11 + 07 + 05`
- 重构完成后验收：`00 + 10`

---

## 场景自检

1. [ ] 已明确这是重构，不是新功能或顺手修逻辑
2. [ ] 已给出风险等级、提交切分和回滚思路
3. [ ] 已说明哪些行为必须保持不变
4. [ ] 涉及核心链路或公共符号时，会先过方案关卡
5. [ ] 重构结论会沉淀到分析报告或架构知识库
