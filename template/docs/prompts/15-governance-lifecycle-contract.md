# 15. Governance Lifecycle Contract（共享治理生命周期合同）

> 本文件不是独立场景，也不是独立阶段。
> 它定义所有活跃运行入口都必须继承的共享治理合同，重点解决“开始当前轮次前该做什么”和“准备结束当前轮次前必须经过哪些确认”。
> GitHub Copilot / `.github/prompts/*.prompt.md` 应自动注入本文件；手工 `#file` 模式下，若希望严格对齐运行骨架，也应显式带上本文件。

## 定位

治理层负责统一约束以下行为：

- 最新团队规范获取
- 任务文档回查与多轮补充处理
- 规划标记与完成标记
- 待确认点批量汇总与确认协议
- tmux-first 执行纪律
- 结束前用户确认与默认“继续检查文档”

这些规则属于共享生命周期，不应散落在某个 Scene、Stage 或 Method 里单独维护。

基础治理能力默认由 `wms-task-governance` 承担。

当前任务文档（通常是 `docs/tasks/.../README.md`）是本轮协作的治理控制面：

- 首轮进入时要完整阅读一次
- 后续每轮结束前要做增量补充检查
- 规划标记、处理结果、完成标记都以它为主事实源
- 若分给 subagent，也要把它当成必须继承的控制面一起传递

## Governance Subcontracts（治理子合同）

### 1. Policy Sync Contract

- 每轮开始先通过团队自己的规则源获取最新规范
- 若委派给 subagent，必须同步告知“先读最新规范，再进入具体任务”
- 若规则源与本地 Prompt 冲突，以规则源最新版本为准

### 2. Task Record Contract

- 首轮先完整阅读当前任务文档，再写本轮 `规划任务:` 标记
- 完成本轮有效产出后，先写 `任务处理结果:`，再决定是否允许写完成标记
- 所有关键结论优先写回任务文档及其 `artifacts/`

### 3. Delta Check Contract

- 首轮完整阅读控制文档一次
- 后续默认按“上一次规划标记之后的增量 + 文档尾部”检查补充需求
- 若发现 `【补充】`、`【更新】`、新增 TODO 或尾部新增要求，必须重新规划

### 4. Human Confirmation Contract

- 结束前必须显式询问：`本文档中的任务是否已经处理完毕`
- 若当前平台不支持真正的输入框 / 确认框，则翻译成“显式发问 + 默认继续检查文档”
- 未收到用户允许收尾的确认前，不得进入完成态

### 5. Delegation Contract

- 分给 subagent 时，至少同步：
  - 当前控制文档路径
  - 当前轮次规划摘要
  - 本轮目标
  - 非目标 / 禁止事项
  - 完成标准与回写位置
- 不允许把共享治理合同留在主 agent 自己身上，而让 subagent 自由发挥

## Start Contract（开始合同）

每次进入任一活跃入口前，都必须先执行：

1. 通过团队自己的规则源获取最新规范，例如：`curl "<your-latest-engineering-rules-url>"`
2. 检查任务目录与 `README.md` 是否已存在；不存在则先初始化，存在则把它当作治理控制面
3. 首轮进入当前任务时完整阅读控制文档；续做任务先按上轮规划标记之后的增量和文档尾部检查是否有 `TODO`、`【补充】`、`【更新】`
4. 为本轮写入新的 `规划任务:` 标记；若是补充需求，则写成 `第x次补充`
4. 如涉及 shell / 构建 / 测试 / 批处理命令，优先使用 `tmux`
5. 进入具体 Scene / Strategy / Method / Gate 前，先确认本轮边界、控制文档路径和默认产物

## Finish Contract（结束合同）

准备结束当前轮次前，都必须执行：

1. 回查 `README.md` 与当前对话，确认是否存在新的补充要求、待办项或未沉淀结论
2. 该落盘的内容必须先落到 `docs/tasks/`、`artifacts/`、`business-insights.md` 等位置
3. 涉及代码 / SQL / 配置改动时，必须先经过评测门禁；未通过不得写完成标记
4. 必须显式询问用户：`本文档中的任务是否已经处理完毕`
5. 若当前平台不支持真正的输入框 / 确认框，则默认按“需要继续检查文档”处理，而不是默认结束
6. 若本轮出现必须人工确认的阻塞点，必须在当前线程一次性汇总全部必须确认项，并由平台适配层承接确认协议
7. 用户确认结果默认应在同一线程继续推进，不要求切换 prompt 或重开会话
8. 文档写入后要再次做增量补充检查；若尾部出现新要求，必须回到新一轮规划，而不是直接结束
9. 只有在确认无新增要求、允许完成、且用户确认可以收尾后，才进入完成态或写完成标记

## Platform Note（平台适配）

- 在 GitHub Copilot Chat 中，无法强制弹出阻塞式输入框，因此必须把“结束前确认”翻译成显式发问 + 默认继续检查文档
- 在不同平台中，不要把 `askQuestions` 这类具体工具名写成治理本体；治理层只定义“确认协议”，具体工具由适配层决定
- 在支持 `askQuestions` 的 VS Code / IDE 适配层中，应由适配层调用该能力来承接 `Human Confirmation Contract`
- 在支持交互式输入框的运行平台中，应优先使用确认框，并把默认选项设为“需要继续检查文档”
- 若你的私有项目额外引入了 watcher / 守卫脚本，那属于维护者自定义实现，不是本公开模板的默认主流程

## 使用要求

- `Governance` 决定“这一轮开始和结束的硬边界”
- `Orchestrator` 负责决定当前公开给用户的 `DISCOVER / DELIVER / VERIFY`
- `Scene` 决定“任务是什么”
- `Method` 决定“是否叠加专项方法”

模式（`TDD / ACD / AOD`）属于 **Orchestrator 之下的执行剖面**，不是新增架构层级。
