# 提示词 14 — 统一执行模式目录

> 本文件是“执行模式”的唯一事实来源。
> 模式不是第六层架构，而是挂在 **Orchestrator** 之下的执行剖面，用来回答“这轮更适合怎么推进”。

---

## 总规则

1. 每轮只允许一个主模式，避免模式之间互相打架
2. 模式优先级由总控自动判定；若用户明确指定模式，以用户指定为准
3. 模式解决的是“推进方式”，不是“任务场景”
4. 若模式不再适合当前任务，必须显式说明切换原因

---

## Mode 1：TDD

- 对应入口：`/wms-tdd`
- 对应 Prompt：`17-tdd-mode.md`
- 对应 Skill：`tdd-mode`

### 适用信号

- 验收标准明确
- 可以被测试、断言或回归点很好描述
- 希望先收敛失败信号，再做最小实现

### 核心做法

- 先把验收标准转成测试点 / 断言 / 失败信号
- 先说明要补哪些测试，再说明最小实现
- 实现后回到评测门禁，验证回归结果

### 不适合

- 线上异常证据不足
- 还在讨论架构取舍
- 任务本质是链路观察或补证

---

## Mode 2：ACD

- 对应入口：`/wms-acd`
- 对应 Prompt：`18-acd-mode.md`
- 对应 Skill：`acd-mode`

### 适用信号

- 跨模块 / 跨服务 / 公共接口 / 数据库结构
- 方案存在明显取舍
- 需要先确认边界、契约、回滚和风险

### 核心做法

- 先明确架构目标和约束
- 再列候选方案、取舍理由、回滚思路
- 人工确认前，不进入实现

### 不适合

- 只是一个小范围、验收标准明确的改动
- 只是补证或线上排障

---

## Mode 3：AOD

- 对应入口：`/wms-aod`
- 对应 Prompt：`19-aod-mode.md`
- 对应 Skill：`aod-mode`

### 适用信号

- 现象很多，但证据不足
- 线上异常、数据不一致、链路不清
- 需要日志、时间线、调用链、证据链和假设收敛

### 核心做法

- 先盘点已知证据、缺失证据和下一步验证动作
- 明确止血、影响范围和假设验证路径
- 证据不足时不把猜测写成结论

### 不适合

- 需求和验收已经清楚，可直接进入测试驱动
- 方案取舍才是主要问题

---

## 模式切换规则

- `AOD -> ACD`：证据已足够，开始讨论架构和方案取舍
- `ACD -> TDD`：方案已明确，验收标准可测试，允许进入实现
- `AOD -> TDD`：异常已定位、修复目标清楚，并且可以先写回归测试

---

## 输出协议

若本轮启用了模式，输出中至少包含：

```markdown
## 执行模式
- 当前模式：
- 选择原因：
- 这轮的推进重点：
- 何时需要切换模式：
```
