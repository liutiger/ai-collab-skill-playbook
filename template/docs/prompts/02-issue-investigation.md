# 提示词 02 — 线上问题排查场景检查包

> 对话开头可引用 `#file:docs/prompts/02-issue-investigation.md`。
> 更推荐先走 `#file:docs/prompts/11-scene-router.md` 判定场景，再决定是否叠加 `09-plan-gate.md`、`08-link-confirmation.md` 或 `07-auto-dev-orchestration.md`。
>
> **兼容入口**：本场景的权威工作法以 `issue-investigation-scene` 为准。
> 这是场景检查包，不承担完整共性流程。

---

## 何时使用

- 线上异常、日志报错、数据不一致、业务链路中断
- 需要先止血、梳理时间线、做根因分析和补偿判断
- 用户诉求是“先查清楚发生了什么”，不是直接做新功能

## 不用于

- 只是确认真实业务链路：优先 `08-link-confirmation.md`
- 已知修复方案且只剩实现：再进 `07-auto-dev-orchestration.md`
- 只做代码审查：改走 `03-code-review.md`

---

## 场景判定信号

- 常见用户表达：`线上异常`、`broken pipe`、`为什么没更新`、`数据不一致`
- 常见产出：时间线、影响范围、止血方案、根因、补偿方案、hotfix 结论

---

## 专项检查重点

### 1. 时间线和影响范围

- 故障时间精确到分钟
- 影响对象精确到单据、SKU、仓库、接口或任务批次
- 区分“用户反馈”与“系统证据”

### 2. 止血优先

- 是否需要回滚、开关降级、暂停消费、数据隔离
- 即使结论是“不需要止血”，也要说明依据

### 3. 5W 根因分析

- 直接现象
- 上层原因
- 深层原因
- 代码 / 配置 / 数据根因
- 为什么没有被提前发现

### 4. 数据补偿与后续修复

- 是否需要 SQL / 重放 / 重新消费 / 手工补单
- hotfix 与根本修复分开描述

---

## 默认产出物

- `artifacts/hotfix-changes.md`
- `artifacts/data-fix-*.sql`
- `docs/tasks/.../README.md`
- `business-insights.md`

---

## 推荐组合

- 先排范围与方案：`00 + 11 + 09 + 02`
- 先确认真实链路：`00 + 08`
- 确认方案后落地修复：`00 + 11 + 07 + 02`
- 修复后验收：`00 + 10`

---

## 场景自检

1. [ ] 已先确认时间线、影响范围和止血判断
2. [ ] 没有在证据不足时直接拍脑袋给根因
3. [ ] 已保留数据补偿视角，而不是只盯代码
4. [ ] 需要 hotfix 时，会在修复后进入评测门禁
5. [ ] 排查结论会沉淀到 `docs/tasks/`
